# kubeadm Cluster Setup

This file documents the kubeadm steps used to create the Kubernetes cluster.

## Control Plane Initialization

The control plane was initialized on `k8s-master`.

```bash
sudo kubeadm init \
  --apiserver-advertise-address=172.31.12.17 \
  --pod-network-cidr=192.168.0.0/16
```

## Why use --apiserver-advertise-address?

The API server should advertise the private IP of the master node so worker nodes inside the AWS VPC can communicate with it internally.

## Why use --pod-network-cidr?

The pod network CIDR must match the CNI plugin configuration.

For this lab, Calico was used with:

```text
192.168.0.0/16
```

## Configure kubectl

After kubeadm initialized the control plane, kubectl was configured for the `ubuntu` user:

```bash
mkdir -p $HOME/.kube
sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
sudo chown $(id -u):$(id -g) $HOME/.kube/config
```

## Initial Node Status

Before installing a CNI plugin, the master node appeared as:

```text
NotReady
```

This is normal.

The node becomes Ready after the pod network is installed.

## Install Calico CNI

```bash
curl -O https://raw.githubusercontent.com/projectcalico/calico/v3.32.0/manifests/calico.yaml
kubectl apply -f calico.yaml
```

## Join Worker Nodes

Worker nodes were joined using the `kubeadm join` command generated by `kubeadm init`.

The actual token is not included in this repository.

Safe format:

```bash
sudo kubeadm join <MASTER_PRIVATE_IP>:6443 \
  --token <TOKEN> \
  --discovery-token-ca-cert-hash sha256:<HASH>
```

## Final Cluster Status

```text
k8s-master     Ready
k8s-worker-1   Ready
k8s-worker-2   Ready
```
